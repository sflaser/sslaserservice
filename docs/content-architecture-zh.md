# SkyFire 网站内容结构方案（中文审稿版）

本文档不直接上线，只用于当前网站内容审稿、页面结构确认，以及后续中英双语实施前的内容基线。

## 1. 网站定位

SkyFire 当前最合适的定位不是“个人专家博客”，也不是“通用工业模板站”，而是：

**面向海外客户的激光维修、定制、组件与技术商务承接平台。**

对应的商业角色分工：
- SkyFire：前台品牌、获客、商务承接、内容包装、国际客户沟通。
- 技术合作方 / 供应侧：技术判断、产品定义、维修执行、供应能力、深度工程支持。

因此，网站内容应突出：
- 专业判断
- 商业可信度
- 应用场景理解
- 维修 / 定制 / 组件 / 询价路径

而不是突出：
- 个人发声
- 敏感技术细节
- 过深的内部工艺知识

## 2. 导航结构（建议架构）

### 主导航
- Home / 首页
- Services / 服务与能力
- Applications / 应用场景
- Products / 产品与备件
- Resources / 资源中心
- About / 关于 SkyFire
- Contact / 联系与询价

### 次级入口
- Blog article / 单篇资源详情页
- Product detail / 单个产品详情页（后续可做）
- Thank you / 提交成功页
- Admin / 后台管理（不作为公开主导航展示）

## 3. 页面级结构（中文对照）

## Home / 首页

首页目标不是承载全部内容，而是完成四件事：
1. 让客户 10 秒内知道你们做什么。
2. 让客户判断你们是否可信。
3. 让客户知道自己属于哪个服务路径。
4. 让客户发起 RFQ。

### 3.1 Hero / 首屏
**作用**：第一屏完成定位。  
**当前英文方向**：维修、定制、组件、技术商务承接。  
**中文审稿重点**：
- 是否足够明确地表达“固体激光维修 + 半导体/定制 + 组件供应”？
- 是否避免了“什么都做”的泛化表达？
- 是否把“商业前台 + 技术后台”讲清楚，而不是讲得太抽象？

**中文建议表达方向**：
- 面向固体激光维修、定制光子项目和关键组件供应的国际业务前台。
- 服务对象包括工业设备商、实验室、研究机构及特殊应用团队。
- 重点不是卖模板式服务，而是帮助客户更快判断“修、换、买、定制”哪条路径更合理。

### 3.2 Engagement Patterns / 客户常见进入场景
**作用**：替代假的五星好评，改成真实客户需求模式。  
**中文审稿重点**：
- 是否真实反映客户来找你们的原因？
- 是否避免了夸张、不可信的 public testimonial 风格？

**建议保持的主题**：
- 故障恢复
- 老旧平台 / 停产型号支持
- 配件优先采购
- 定制项目启动
- 应用匹配判断
- 商务前台承接复杂项目

### 3.3 Capabilities / 服务与能力
**作用**：明确三大主线。  
**建议长期固定为**：
- Solid-State Laser Repair / 固体激光维修
- Custom Laser Manufacturing / 定制激光制造与项目协作
- Components and Spare Parts / 组件与备件供应

**中文审稿重点**：
- 每项服务是否足够聚焦？
- 是否避免了服务边界混乱？
- 是否说明了客户何时该点击 RFQ，何时该直接邮件沟通？

### 3.4 Repair Workflow / 维修流程与案例视频
**作用**：证明这不是空洞营销。  
**中文审稿重点**：
- 视频标题是否足够专业？
- 是否避免“课程感/培训感”太重？
- 是否更偏“实际维修能力展示”？

### 3.5 Comparison / 决策比较区
**作用**：回答“为什么找 SkyFire”。  
**当前逻辑**：不做模板式优劣表，而是围绕：
- 诊断深度
- 商务分流能力
- 平台熟悉度
- 后续延展能力

**中文审稿重点**：
- 是否符合真实商务逻辑？
- 是否能让客户理解：你们不是普通维修商，而是更擅长复杂案件的商业前台？

### 3.6 Supported Systems / 已支持平台
**作用**：这不是电商目录，而是“平台熟悉度证明”。  
**中文审稿重点**：
- 是否容易让客户误解为“公开售卖目录”？
- 是否应强调“这是经验覆盖，不是完整 catalog”？
- 平台说明是否更像“应用和熟悉度”，而非简单堆品牌名？

### 3.7 Applications / 应用场景
**建议保持的三大块**：
- Display Manufacturing / 显示制造
- Semiconductor Tools / 半导体设备
- PCB Precision Processing / PCB 精密加工

**中文审稿重点**：
- 是否对应你们真实客户类型？
- 是否需要补充 Research / Lab 入口（可以作为后续扩展，不一定放首页主 tab）？

### 3.8 About SkyFire / 关于 SkyFire
**方案选择：A（已采用）**

这块更适合作为：
- 技术范围说明
- 商业模式说明
- 合作方式说明
- 内容边界说明

**不建议写成**：
- 公司宣传稿
- “我们什么都能做”
- 过于泛化的行业宏大叙事

**中文应表达的重点**：
- SkyFire 是国际市场前台，不是单纯维修门店。
- 真正的技术深度来自维修经验、供应协作和项目支持网络。
- 网站公开内容以“客户理解和商务沟通”为目标，不公开敏感 know-how。

### 3.9 Products / 产品与备件
**作用**：承接可购买、可询价、可定制的产品需求。  
**中文审稿重点**：
- 是否区分清楚：直接购买 / 先询价 / 定制讨论？
- 产品卡片是否更像工程产品而不是普通电商？

### 3.10 Featured Resources / 精选资源
**作用**：首页只放入口，不放完整资源库。  
**中文审稿重点**：
- 首页展示 3 篇精选即可。
- 作用是建立专业感，不是让首页无限加长。
- 完整资源应进入独立页 `/resources.html`。

### 3.11 Contact / RFQ / 联系与询价
**作用**：完成转化。  
**当前方向正确**：
- 结构化表单
- 可上传文件
- 可直接提交 RFQ

**中文审稿重点**：
- 表单字段是否对技术判断真的有帮助？
- 是否仍有可以删减的无效字段？
- 手机和平板宽度下是否足够轻松可填？

## Resources / 资源中心

### 核心定位
资源中心不应该是“个人博客”，而应该是：

**面向买家和技术使用方的知识中心。**

### 适合长期积累的 4 类内容
1. Maintenance Guides / 维护指南
2. Application Notes / 应用说明
3. Selection Guides / 选型与采购判断
4. Commercially Safe Case Patterns / 商务安全型案例

### 不建议重点做的内容
- 太个人化的发声
- 太多行业评论
- 太深的维修内部工艺
- 可复制的技术诀窍
- 太敏感的供应链细节
- 看起来像“我们什么都能做”的长篇宣言

### 当前资源页角色
- 首页：只放精选 3 条 + 查看全部
- 资源页：承担长期累积
- 单篇详情：承担 SEO、专业感、RFQ 引导

## Blog article / 单篇资源详情页

单篇页面的目标不是“让人读完觉得你很厉害”，而是：
- 读完后更信任你们
- 读完后更知道自己该问什么
- 读完后愿意发 RFQ

### 单篇文章建议固定结构
- H1 标题
- 一段英文摘要 / Excerpt
- 3-5 个 H2 结构
- 必要时 H3
- 结尾 CTA：
  - Request a Quote
  - Ask About Compatibility
  - Send Model Details

## About / Platform Model / 关于与平台模式

这部分建议未来作为单独页面继续展开，但目前首页已有承载。

### 推荐内容方向
- 我们服务哪些客户
- 我们解决什么问题
- 我们如何与技术合作方协作
- 什么内容公开，什么内容不公开
- 为什么这种“前台品牌 + 技术后台”模式更适合复杂激光项目

## 4. 中文审稿工作流（当前最适合）

在真正做多语种上线前，建议先按下面流程走：

1. 保持前端现阶段以英文为主。
2. 用本文档作为中文审稿基线。
3. 每新增一个区块或文章，先写：
   - 中文定位说明
   - 英文正式文案
4. 中文主要用于内部确认，不一定立即公开上线。

这样做的好处：
- 郑老师更容易判断内容是否准确。
- 你可以先确认方向，再决定要不要做真正的中文前台。
- 可以避免过早上双语，导致维护成本翻倍。

## 5. 后续如果要做真正双语，建议的技术架构

暂时不直接上线，但建议后续按这个思路：

### 方案建议
- 英文主站：默认 `/`
- 中文站：`/zh/`

### 页面映射示例
- `/` -> 英文首页
- `/zh/` -> 中文首页
- `/resources.html` -> 英文资源页
- `/zh/resources.html` -> 中文资源页
- `/blog.html?slug=...` -> 英文文章详情
- `/zh/blog.html?slug=...` -> 中文文章详情

### CMS 层后续建议
博客和产品数据最终可以扩展为：
- `title_en` / `title_zh`
- `excerpt_en` / `excerpt_zh`
- `content_en` / `content_zh`
- `short_description_en` / `short_description_zh`
- `full_description_en` / `full_description_zh`

但这一步现在不急。

## 6. 当前两篇资源内容的定位建议

### 文章 1：维护类
《Preventive Maintenance Guide for Solid-State Lasers: 8 Practical Actions to Reduce Downtime》

判断：**适合保留**。

原因：
- 紧贴维修业务
- 客户能看懂
- 风险低
- 有利于引导询盘

### 文章 2：业务范围总述类
那篇覆盖固体激光维修、半导体激光定制、量子、科研、供应链等内容的长文。

判断：**不适合作为普通博客原样发布**，更适合归入 `About / Platform Model`。

原因：
- 内容过大过全
- 更像能力说明而不是资源文章
- 转化路径不够聚焦
- 对外会显得范围太大

建议：
- 拆入 About / Platform Model
- 或拆成 3 篇垂直资源

## 7. 图片更新的审稿方式

后面拍摄新图片时，建议每张图都按下面 4 项给内部备注：
- 放在哪个区块
- 主要想表达什么
- 哪些不能裁
- 哪些图优先级最高

如果内部没有时间细分，也可以先批量给图，再由我做统一编排和替换。

## 8. 当前阶段的结论

当前网站内容方向已经从“模板站”转到较清晰的结构：
- 首页：商业定位 + 能力分流 + 询价入口
- 资源：专业感 + SEO + 询价质量提升
- About：承载平台模式和技术协作逻辑
- RFQ：完成真正转化

在没有正式上线中文之前，最合理的做法就是：

**用中文做内部审核，用英文做外部表达。**
